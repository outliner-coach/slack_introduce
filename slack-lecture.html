No-Code

🏁 슬랙 워크플로란?

정의

코딩 지식 없이, 슬랙 내에서 반복적인 업무를 자동화하는 '레고 블록' 같은 기능입니다.

  • 누가 사용하나요? 개발자가 아니어도 누구나 만들 수 있습니다.
    • 기획자, 마케터, PM, 운영 담당자 등
  • 핵심 구성:
    • 트리거 (Trigger): 워크플로를 "시작"시키는 조건
      (예: 특정 시간, 이모지 클릭, 채널 입장)
    • 액션 (Action): "수행"할 동작
      (예: 메시지 보내기, 폼 전송, 구글시트 추가)
  • 위치: 슬랙 메뉴 > 도구 > 워크플로 빌더

핵심 개념

🧩 워크플로의 핵심: '변수' 활용

'변수'란?

이전 단계의 정보를 담아 다음 단계로 전달하는 '데이터 보관함'입니다.

예시: 사용자가 폼에 입력한 '이름'을 채널 메시지에 "OOO님이 휴가를 신청했습니다."라고 표시할 때 사용.

활용 방법 (3단계)

1️⃣ 데이터 생성
(트리거)
→
2️⃣ 변수 삽입
(액션)
→
3️⃣ 메시지 전송
(결과)
Step 1: '폼 제출' 트리거에서 '이름' 입력란을 만듭니다.
Step 2: '메시지 보내기' 단계에서 '변수 삽입' 버튼을 클릭.
Step 3: '폼에서 응답한 사람', '이름' 입력값 등 원하는 변수를 선택하여 메시지 본문에 추가합니다.

🚧 슬랙 워크플로의 한계

워크플로는 "정해진, 간단한" 자동화에 강력합니다. 하지만...

  • ❌ 복잡한 로직 불가
    • IF... ELSE... (만약 A이면 B, 아니면 C) 같은 조건 분기나 반복(Loop)이 어렵습니다.
  • ❌ 양방향 상호작용의 한계
    • 봇처럼 사용자와 대화를 주고받거나, 버튼 클릭 시 원본 메시지를 수정하는 등 세밀한 제어가 불가능합니다.
  • ❌ 제한된 연동
    • 구글시트, Jira 등 기본 연동 앱 외에 사내 DB나 복잡한 외부 API 연동은 불가능합니다.

Q. 더 복잡하고 유연한 자동화가 필요하다면?

A. ➡️ Part 2. 슬랙봇 (Pro-Code)이 필요합니다!

💻 알아두면 좋은 기본 개념

슬랙봇을 이해하기 위해 필요한 3가지 핵심 개념

👤

클라이언트 (Client)

사용자가 보는 화면

웹 브라우저, 슬랙 앱 등 우리가 직접 사용하는 프로그램입니다.

예시: 슬랙 메신저 화면

🖥️

서버 (Server)

뒤에서 일하는 컴퓨터

요청을 받아서 처리하고 결과를 돌려주는 컴퓨터입니다. 24시간 켜져 있습니다.

예시: 슬랙봇 프로그램이 실행되는 곳

📊

데이터베이스 (Database)

정보를 저장하는 창고

데이터를 체계적으로 저장하고 필요할 때 꺼내 쓸 수 있게 해주는 시스템입니다.

예시: 구글시트, MySQL 등

🔗

이들의 관계

어떻게 연결되나요?

① 클라이언트가 서버에 요청 → ② 서버가 데이터베이스에서 정보 조회 → ③ 서버가 클라이언트에 결과 전송

예시: 투표 버튼 클릭 → 서버 처리 → 결과 업데이트

제작 프로세스

🛠️ 봇 만드는 법 (6단계)

1. [Slack] 앱 생성
api.slack.com > Your Apps > Create New App (From scratch)
2. [Slack] 기능 선택
만들고 싶은 봇의 유형에 따라 필요한 기능을 활성화합니다. (다음 슬라이드)
3. [Server] 로직 개발
선택한 기능을 처리할 서버 코드를 작성합니다.
4. [Slack] 권한(Scope) 설정
봇이 행동(메시지 쓰기, 파일 읽기 등)할 수 있도록 OAuth & Permissions에서 권한을 부여합니다.
5. [Slack] 앱 설치
워크스페이스에 봇을 설치하고 봇 유저 토큰(xoxb-...)을 발급받습니다.
6. [Server] 토큰 적용
발급받은 토큰을 서버 코드에 적용하여 연동을 완료합니다.

핵심 기능

⚙️ 봇의 4가지 핵심 기능

봇이 "무엇을" 하느냐에 따라 필요한 기능(설정)이 다릅니다.

📨

1. Incoming Webhook

기능: 외부(내 서버)에서 슬랙 채널로 메시지를 보내기만 할 때 (단방향)

예시: 시스템 장애 알림, 배포 완료 알림

⌨️

2. Slash Commands

기능: 사용자가 /명령어를 입력하여 봇을 호출할 때

예시: /배포 [서비스명], /회의실 예약

👂

3. Event Subscriptions

기능: 봇이 채널의 이벤트를 항상 듣고(Listen) 반응할 때

예시: 특정 키워드 감지, 이모지 반응, 채널 입장 감지

🔑

4. Bot User Token

기능: 봇이 스스로 행동(메시지 쓰기, 읽기, 수정)하기 위한 만능 열쇠

필수: 위 2, 3번 기능을 사용하거나, 봇이 능동적으로 메시지를 보낼 때 반드시 필요

FAQ

🕵️‍♀️ 봇 유저 토큰 vs 유저 토큰

구분 Bot User Token (봇 토큰) User Token (유저 토큰)
시작 문자 xoxb- xoxp-
주체 봇 (앱) 자신 봇을 설치/사용한 특정 유저
권한 봇에게 부여된 Scopes 만큼만 행동
(예: chat:write, reactions:read)		 해당 유저가 가진 모든 권한을 위임받아 행동
(유저의 비공개 채널 접근, 유저 대신 글쓰기)
특징 사용자와 무관하게 봇이 독립적으로 작동 매우 강력하고 민감한 권한
용도 99%의 일반적인 봇 활동
(알림, 응답, 키워드 감지 등)		 유저 개인을 대신하는 작업
(개인 캘린더 연동, 개인 DM 정리)

💡 결론

특별한 경우가 아니면 항상 봇 유저 토큰(xoxb-)을 사용해야 합니다.

📌 프로젝트 개요

구글시트에서 간단히 설정하면 슬랙에 투표가 자동 생성됩니다.

🎯

해결하려는 문제

• 매번 저녁 메뉴 결정할 때 대화가 길어짐
• 누가 뭘 선택했는지 파악 어려움
• 마감 시간 관리가 수동적

✨

제공하는 해결책

• 버튼으로 간편하게 투표
• 실시간으로 결과 확인
• 자동 마감 처리

사용된 기술 스택

🛠️ 서버: Google Apps Script (무료 클라우드 서버)

🔑 슬랙 기능: Bot User Token (메시지 전송/수정), Event Subscription (버튼 클릭 감지)

사용자 플로우

1️⃣ 관리자가 구글시트에 투표 정보 입력
(제목, 선택지, 마감시간, 채널)
↓
2️⃣ 체크박스 클릭 → 슬랙 투표 메시지 자동 생성
↓
3️⃣ 팀원들이 버튼 클릭하여 투표
↓
4️⃣ 실시간 결과 업데이트 (참여자 이름 표시)
↓
5️⃣ 마감 시간 되면 자동으로 투표 종료

1️⃣ onEdit 트리거 (투표 생성)

• 고유한 pollId 생성으로 중복 투표 방지
• 슬랙 chat.postMessage API 호출하여 투표 메시지 전송

2️⃣ doPost 웹훅 (버튼 클릭 처리)

• 슬랙에서 버튼 클릭 시 스크립트의 doPost 함수가 호출됨
• LockService로 동시 접근 제어 (충돌 방지)
• 구글시트에서 투표 결과 JSON 데이터 조회/업데이트
• 중복 투표 시 기존 투표 취소 로직 포함
• 슬랙 chat.update API로 원본 메시지 수정 (실시간 업데이트)

3️⃣ checkDeadlines 트리거 (자동 마감)

• 시간 기반 트리거로 주기적 실행 (예: 1분마다)
• 모든 투표의 마감 시간 확인
• 마감된 투표는 버튼 제거 및 "🔒 마감" 표시 추가
• 시트에 "[마감됨]" 태그 추가하여 재처리 방지

📦 작가냥 발주 - 작동 원리 상세

n8n 노드별 역할

1️⃣ Schedule Trigger

• 매일 오전 8시에 워크플로 자동 시작
• 사람이 없어도 알아서 실행되는 "트리거"
• 워크플로의 시작점

2️⃣ JavaScript Code Node (날짜 생성)

• 오늘 날짜 - 1일 = 어제 날짜 계산
• YYYY-MM-DD 형식으로 포맷팅
• 이 날짜를 다음 노드로 전달

3️⃣ HTTP Request (Redash API 호출)

• Redash에 "어제 데이터 조회해줘" 요청
• POST → Redash 쿼리 실행 시작
• job_id 받아옴 (처리 진행 중 ID)

4️⃣ Wait Node (대기)

• Redash가 데이터 처리할 시간 부여
• 너무 빨리 결과 요청하면 아직 준비 안됨
• 일정 시간 대기 후 다음 단계로

5️⃣ HTTP Request (결과 조회)

• job_id로 결과 조회
• JSON 형태로 데이터 받아옴
• 배열 형태의 데이터 (여러 행)

6️⃣ Code Node (데이터 변환)

• JSON 배열을 개별 항목으로 분리
• 각 행이 별도 데이터가 되도록 변환
• 구글시트에 저장하기 좋은 형태로

7️⃣ Google Sheets Node (저장)

• 변환된 데이터를 구글시트에 자동 저장
• appendOrUpdate 모드: 기존 데이터 유지하고 새 데이터 추가
• user_id 기준으로 중복 확인

8️⃣ Slack Node (알림)

• 지정된 슬랙 채널에 완료 알림
• "○○ 일자 데이터 ○건 스크랩 완료" 메시지
• 구글시트 링크 포함

🔍 왜 n8n을 사용했나요?

• 비주얼 인터페이스: 코드 없이 드래그 앤 드롭
• 다양한 통합: Redash, Slack, Google Sheets 모두 지원
• 에러 처리: 실패 시 재시도, 알림 등 자동 처리
• 디버깅 쉬움: 각 단계별 데이터 확인 가능

⭐ Review Scrapper - 작동 원리 상세

핵심 기능별 설명

1️⃣ triggerReviewScrapers (메인 함수)

• App_List 시트에서 앱 목록 읽기
• 각 앱의 storeType 확인 (google/apple)
• 스토어별 적절한 수집 함수 호출
• 시간 기반 트리거로 자동 실행 (매일 1회)

2️⃣ 구글 플레이스토어 수집 (Playwright)

• 외부 Playwright 서버에 HTTP POST 요청
• appId와 appName 전달
• Playwright가 브라우저로 실제 페이지 접속하여 크롤링
• 수집된 리뷰 데이터를 JSON으로 받아옴
• doPost() 함수로 처리

3️⃣ 애플 앱스토어 수집 (XML RSS)

• 애플이 제공하는 공식 RSS 피드 사용
• XML 형태로 최신 리뷰 제공 (최대 50개)
• XmlService로 파싱하여 데이터 추출
• 최근 10일 리뷰만 필터링

4️⃣ 중복 방지 로직

• Reviews 시트: reviewId로 중복 확인
• Slack_Map 시트: 슬랙 포스팅 이력 관리
• 시트에서 행 삭제해도 재포스팅 안 됨 (Slack_Map 덕분)
• Set 자료구조로 메모리 내 빠른 조회

5️⃣ 슬랙 포스팅

• chat.postMessage로 리뷰 개별 포스팅
• 리뷰 내용을 멀티라인 인용구(>>>)로 표시
• 메시지 TS(timestamp) 저장 → 스레드 참조용
• 개발자 답변 있으면 chat.postMessage로 스레드 댓글 추가

6️⃣ Recency Gating (최근 데이터 필터링)

• LOOKBACK_DAYS (10일) 설정으로 오래된 리뷰 스킵
• 예외: 오래된 리뷰라도 개발자 답변이 최근이면 포함
• MAX_REVIEWS_TO_PROCESS (30개) 한도로 과부하 방지

🔍 사용된 주요 API

• UrlFetchApp.fetch(): HTTP 요청 (Playwright 서버, 슬랙 API)
• XmlService.parse(): 애플 RSS XML 파싱
• SpreadsheetApp: 구글시트 읽기/쓰기
• chat.postMessage: 슬랙 메시지 & 스레드 댓글

⭐ Review Scrapper - 핵심 포인트 & 배운 점

데이터 구조 (3개 시트 활용)

📋

1. App_List 시트

수집할 앱 목록 관리
컬럼: App ID | App Name

2. Reviews 시트

컬럼: reviewId | App ID | App Name | author | date | rating | content | timestamp

3. Slack_Map 시트

슬랙 포스팅 이력 관리
컬럼: reviewId | ts (타임스탬프) | thread
중복 포스팅 방지 핵심 데이터

Thread (스레드) 활용

💬 부모 메시지: 원본 리뷰 (저장된 ts: 1234567890.123456)
↓ thread_ts 참조
💬 스레드 댓글: 개발자 답변 (부모의 ts를 thread_ts로 사용)

→ 리뷰와 답변이 하나의 스레드로 묶여서 보기 편함!

이 사례에서 배운 핵심 개념

✅ 외부 API 연동: Playwright 서버처럼 외부 서비스를 호출하는 방법
✅ XML 파싱: 웹에서 제공하는 다양한 데이터 형식 처리
✅ Thread 댓글: 슬랙 메시지를 스레드로 연결하여 맥락 유지
✅ 데이터 정규화: timestamp 소수점 손실 방지, 문자열 변환 등 세심한 데이터 관리
✅ 멀티 스토어 지원: 구글/애플 각각 다른 수집 방법을 하나의 시스템으로 통합

🔍 공통점 발견!

세 가지 케이스 모두 Google Apps Script (서버 역할)와 구글시트 (데이터베이스 역할)를 활용했습니다.
즉, 별도 서버나 DB 없이 구글 도구만으로 충분히 구축 가능합니다!
